Phase 0 · Design & setup

The schema is the spine: modeling four entities around the Connect contact ID

2026-06-10

What was built

  • Repo skeleton: src/{connector,api,processing,emission,common}, sql/, infra/aws/, tests/, docs/.
  • sql/schema.sql — the four core tables (interactions, media_records, transcripts, signals) with inline rationale, constraints, indexes, and an updated_at trigger.
  • .gitignore + .env.example — secret-safety guardrails before any code that could leak one exists.
  • README.md — architecture diagram, data-model summary, reliability and HIPAA/BAA posture.

Why this design

Audio (S3) and metadata (contact trace records) arrive separately and out of order. The central decision was to make the Connect contact_id the business key on every table, so the two halves can be reconciled after the fact instead of assuming one combined payload. interactions is an anchor row that whichever path arrives first creates and the second enriches (upsert).

Alternatives considered: (a) a single wide "calls" table — rejected, it forces audio and metadata to arrive together; (b) a natural-key-only PK on contact_id — rejected in favor of surrogate id + UNIQUE(contact_id) for stable internal references plus a clean dedup key.

How it works

interactions (1) ──< media_records (1) ──< transcripts (1) ──< signals (many)
     contact_id          idempotency_key         redacted_text       signal_type
                         (UNIQUE → no
                          double-ingest)

Idempotency is enforced by the database: media_records.idempotency_key is UNIQUE, so a replayed recording fails the INSERT rather than relying on the app to remember to check. The status column (discovered → queued → ingested → transcribed → failed) is the pipeline's state machine and the basis for every monitoring query.

New concepts learned

Contact Trace Record (CTR): Amazon Connect's metadata record for a contact — queue, agent, timestamps, disconnect reason — emitted separately from the audio. Like a webhook event describing the call, arriving on its own channel.
Database-enforced idempotency: a UNIQUE constraint turns "don't double-ingest" from a hopeful app check into a guarantee the database keeps, even under retries and replayed queue messages.
Redact-before-persist: raw transcript text is held only in memory long enough to redact; only redacted_text is written. raw_text stays NULL, documenting the production design (encrypted-at-rest, access-restricted).

Interview talking point

"Recordings and metadata arrive on separate channels and out of order, so I keyed the whole model on the Connect contact ID and made the interaction row an upsert anchor — whichever side lands first creates it, the second enriches it. Idempotency is a UNIQUE constraint on the media record, so a replayed recording is rejected by the database, not by application logic I have to trust."
