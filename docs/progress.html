Phase 3 · Mock ingestion API ("AcxAPI" clone)

The front door: an authenticated upload API that validates, dedupes, and matches metadata

2026-06-13

What was built

  • A FastAPI service in src/api/ — the receiving side of the ingestion contract. main.py wires three endpoints: GET /healthz (liveness + a DB round-trip, no auth), POST /v1/recordings (the upload), and GET /v1/recordings/{id} (look up a record).
  • Token auth (auth.py): a FastAPI dependency that requires Authorization: Bearer <ACX_API_TOKEN>, compared in constant time with hmac.compare_digest, and fails closed if no token is configured.
  • Codec validation (audio.py): two layered checks — the declared multipart Content-Type (→ 415) and the actual bytes parsed as PCM WAV via the stdlib wave module (→ 400). The byte check also yields duration for free.
  • Idempotent write + metadata matching (db.py): an upsert_interaction that enriches the anchor with CTR-style fields using COALESCE(EXCLUDED.x, …) (fill blanks, never erase), then insert_media_record with ON CONFLICT (idempotency_key) DO NOTHING returning created vs duplicate.
  • A push client (upload_client.py) that POSTs a WAV (real file or a synthetic silent one) with metadata, and a pytest suite (tests/test_api.py) covering auth, idempotency, enrichment-without-clobber, and every error code. Plus sql/validation/phase3_checks.sql.

Why this design

Phase 2's connector wrote to Postgres directly — a deliberate stand-in until the real front door existed. Phase 3 builds that door so we have both sides of the ingestion contract: something that pushes data in, and an API that authenticates, validates, and decides accept/reject with clear status codes. I chose a multipart file upload (audio bytes + metadata) over a JSON S3-pointer because it's the realistic vendor-API shape and it actually exercises codec validation and file handling — the parts the role screens for.

Idempotency here is content-addressable: the key is sha256(bytes), so the same audio is the same recording no matter who uploads it or how often. That feeds the very same UNIQUE column the connector keys on with sha256(bucket/key) — two paths, one dedup authority in the database. I deliberately don't persist the audio bytes in this API: they already live in S3 from Connect, and re-storing them would add cost and a second copy to secure; a comment marks where production would stream to encrypted-at-rest object storage.

How it works

POST /v1/recordings  (multipart: file=<wav> + contact_id, queue, agent, …)
  │  require_token  ───────────────► 401 if bad/missing Bearer token
  │  uuid(contact_id) ─────────────► 422 if not a UUID
  │  Content-Type in {audio/wav…} ─► 415 if wrong label
  │  validate_wav(bytes)  ──────────► 400 if not readable PCM WAV
  ▼
  upsert interactions(contact_id, …)        ← enrich, COALESCE (no clobber)
  INSERT media_records … ON CONFLICT DO NOTHING
        ├─ row returned  ─► 201 created   (new recording)
        └─ no row        ─► 200 duplicate (same bytes seen before, same id)

Validation runs before any DB write, so a rejected upload leaves nothing behind. The happy path commits once; the per-request connection rolls back on any error. The status-code discipline (201 vs 200 vs 401/415/400/422/404) is the whole point — a caller can act on the response without parsing a body.

New concepts learned

Multipart upload + FastAPI Form/File: the audio file and its metadata travel together in one multipart/form-data body; python-multipart lets FastAPI split them into typed parameters.
Layered validation (415 vs 400): the Content-Type is the label on the envelope (cheap, client-declared); parsing the bytes is opening it. Trusting only the label is how bad files slip in, so we do both and return different codes.
Constant-time secret comparison: hmac.compare_digest avoids leaking how many leading characters of the token matched via response timing.
Content-addressable idempotency: hashing the bytes means identity comes from the content itself, so retries and re-uploads collide on the UNIQUE key and become safe no-ops — the DB is the dedup authority, not app logic.
COALESCE upsert (enrich, don't erase): on conflict we set each column to COALESCE(EXCLUDED.col, existing.col), so a later partial update fills blanks without wiping data an earlier one provided — exactly right when metadata arrives in pieces.

Interview talking point

"The ingestion API is the receiving half of the contract: a Bearer-token-authenticated multipart upload that validates the codec at the door — checking both the declared Content-Type and the actual WAV bytes, with distinct 415 and 400 responses — then writes idempotently. The idempotency key is a hash of the audio content, so re-uploads collide on a UNIQUE constraint and return 200 instead of creating a duplicate. The same request also matches the call's metadata onto the interaction with a COALESCE upsert that enriches without clobbering, because audio and metadata arrive separately. It's the same auth-and-idempotency discipline I built on Ignition webhook ingestion, expressed as a clean HTTP contract with deliberate status codes."

Phase 2 · Capstone — backfill

Forward-only events leave history stranded: a backfill scanner for the onboarding motion

2026-06-12

What was built

  • Refactored the connector so the event path and the backfill path share ONE core, ingest_object(bucket, key) in src/connector/consumer.py — parse contact_id → HeadObject → idempotent media_record insert. The live path (process_record) now just unwraps the S3 event envelope and decodes the URL-encoded key, then calls the shared core.
  • src/connector/backfill.py — lists the recordings prefix with a paginator and feeds every existing object through ingest_object. Runs as the least-privilege connector user via its prefix-scoped s3:ListBucket grant.

Why this design

S3 event notifications are forward-only — they fire only for objects created after the notification exists (the same property Phase 1's CTR streaming had). The six real Phase-1 recordings predated the wiring, so no event ever came for them; they sat stranded in S3. That's not an edge case — it's the customer-onboarding motion: when you forward-deploy to a new contact center, their bucket is already full of history. "Event-driven for new conversations, backfill for the history that predates us."

The deliberate choice was to make backfill reuse the exact ingest core rather than copy it, so the two paths can never drift. The one thing that differs is correctly isolated: S3 event keys are URL-encoded (: → %3A) and get decoded in the event wrapper, while ListObjectsV2 returns plain keys — so the shared core always receives a real, decoded key.

How it works

NEW call  ─► S3 event ─► SQS ─► process_record ─(decode key)─┐
                                                             ├─► ingest_object ─► media_record
HISTORY   ─► backfill.py ─► list S3 ─(plain keys)────────────┘        (ON CONFLICT = idempotent)

Because the shared core ends in ON CONFLICT (idempotency_key) DO NOTHING, the backfill is idempotent: objects the live path already ingested return as duplicate no-ops. Proven live — first run: ingested=11, duplicate=2 (the hand-fired object + the synthetic test were already in); second run: ingested=0, duplicate=13, a total no-op. So it's safe to run repeatedly, safe to run alongside live events, and the same scan on a schedule becomes a reconciliation sweep.

New concepts learned

Forward-only notifications: S3 (and Connect CTR) events only cover what's created after wiring; pre-existing data must be pulled in by listing, not waited for.
Backfill vs. queue buffering: connector downtime is handled by SQS retention (messages wait); backfill is for data that never generated an event at all (onboarding history, or a rare missed event caught by reconciliation).
One core, two callers: extracting ingest_object means the live and backfill paths can't diverge — the URL-decode asymmetry is the only difference, and it lives in the event wrapper where it belongs.

Interview talking point

"Event notifications are forward-only, so a new customer's existing recordings never fire an event — I built a backfill scanner that lists the bucket and runs each object through the exact same idempotent ingest the live path uses. First run picked up the history; a second run was a complete no-op because the database UNIQUE constraint makes re-ingestion safe. That's the real onboarding pattern: event-driven for new conversations, a re-runnable backfill for the history — and the same scan on a schedule is my reconciliation check."

Phase 2 · Ingestion connector & queue

The mail carrier: S3 ObjectCreated → SQS → a least-privilege connector that records media idempotently

2026-06-11

What was built

  • Local Postgres via docker-compose.yml (Colima as the Docker engine, no Docker Desktop GUI needed). sql/schema.sql auto-applies on first boot, so the four tables + triggers exist with one command. Teardown is docker compose down -v.
  • SQS main queue + DLQ (infra/aws/provision_sqs.py): standard queue acx-recordings with a redrive policy to acx-recordings-dlq after 5 failed deliveries; 60s visibility timeout; 20s long-poll.
  • S3 → SQS event notification (infra/aws/provision_s3_notification.py): a queue resource policy letting only this bucket in this account send messages, then an s3:ObjectCreated:* notification filtered to the CallRecordings/ prefix (CTR metadata deliberately excluded).
  • Least-privilege IAM user acx-connector (infra/aws/provision_connector_iam.py): scoped to receive/delete on the one queue and s3:GetObject on the recording prefixes — nothing else. Verified it is denied sqs:ListQueues.
  • The connector (src/connector/consumer.py + db.py): long-poll SQS → skip s3:TestEvent → parse contact_id from the filename → HeadObject → upsert interaction anchor → insert media_record at status queued with an idempotency key → commit → delete the message.
  • Test utility + SQL validation: upload_test_recording.py drops a synthetic silent WAV to fire a real event (the DID is released, so no live calls); sql/validation/phase2_checks.sql proves no duplicates / orphans.

Why this design

The queue is the shock absorber between "recordings land whenever" and "we process at our own pace" — the same buffered-ingestion shape as the Ignition webhook handler on my resume, but AWS-native. I chose a standard SQS queue (not FIFO): standard is at-least-once, so it can deliver a message twice, which is exactly why idempotency is enforced one layer down by a UNIQUE constraint rather than trusted in app code. FIFO would add cost and isn't needed when the DB is the dedup authority.

Two credential tiers on purpose: admin creds build the infrastructure (the provisioning scripts), and a separate least-privilege key runs the connector. If the connector key leaked, the blast radius is "read some audio + drain one queue," not the whole account. I provisioned with boto3 scripts rather than console clicks so the infra is reproducible, documented, and teardown-able.

How it works

S3 (recording lands)
  │  s3:ObjectCreated:*  (prefix-filtered to CallRecordings/)
  ▼
SQS  acx-recordings ──(5 failures)──► acx-recordings-dlq
  │  long-poll receive (≤10 msgs)
  ▼
connector  ─ skip s3:TestEvent
           ─ parse contact_id from <contactId>_…wav
           ─ HeadObject (as least-privilege acx-connector)
           ─ upsert interactions(contact_id)        ← anchor
           ─ INSERT media_records … ON CONFLICT DO NOTHING  ← idempotency
           ─ commit  ─► THEN delete SQS message (the ack)

The order is the reliability story. The DB write commits before the SQS delete, so a crash in between just means the message reappears after the visibility timeout and is retried — and the ON CONFLICT makes that retry a harmless no-op. A message that keeps failing is never silently dropped; after 5 tries SQS parks it in the DLQ. Proven live: uploading the same object twice produced one media_record and a duplicate_skip log.

New concepts learned

SQS visibility timeout: on receive, a message is hidden (not deleted) for N seconds. Delete it = done; crash = it reappears and retries. Like an email you mark unread until you've actually finished acting on it.
Redrive policy & DLQ: after maxReceiveCount failed deliveries, SQS moves the message to a dead-letter queue instead of looping forever — a poison message gets quarantined, not lost.
Queue resource policy + confused-deputy guard: S3 is a different service principal, so the queue must explicitly allow s3.amazonaws.com to send — and the aws:SourceArn/aws:SourceAccount conditions ensure only our bucket can, not a stranger's pointed at our queue.
s3:TestEvent: when you attach an S3→SQS notification, S3 sends a one-off test ping to prove it can deliver. The consumer must recognize and skip it.
Two-tier credentials & least privilege: powerful admin keys create resources; a scoped key runs against them. Verified by confirming the connector key is denied ListQueues.

Interview talking point

"S3 fires an ObjectCreated event into an SQS queue, and a connector running as a least-privilege IAM user pulls from it. I lean on SQS's at-least-once delivery on purpose and make ingestion idempotent at the database — a UNIQUE constraint on the media record — so a duplicated message is a no-op, not a double-ingest. The connector commits to Postgres before it deletes the SQS message, so any crash just triggers a safe retry; messages that keep failing land in a dead-letter queue rather than being dropped. It's the same buffered, retry-with-DLQ ingestion pattern I built on Ignition, just AWS-native."

Phase 1 · Amazon Connect source

Standing up the source: live scripted calls → audio + CTR metadata in S3

2026-06-11

What was built

  • Amazon Connect instance acx-practice (us-east-1), with Connect-stored users and both telephony directions enabled. Connect auto-created an S3 data bucket amazon-connect-8ad40e47e8f8 plus a companion -access-logs bucket (S3 server-access logging = an audit trail of who touched the recordings).
  • Recording path: added a Set recording and analytics behavior block (Agent + Customer) to the Sample inbound flow, wired before the transfer-to-queue, and published. Claimed a DID, answered as an available agent in the CCP, and placed a few synthetic, role-played scripted calls → .wav files landed under connect/acx-practice/CallRecordings/.
  • Metadata path: created a Kinesis Firehose stream acx-ctr-stream (Direct PUT → S3, 60s buffer) writing to the connect/acx-practice/CTR/ prefix, then turned on Connect Data streaming for Contact Trace Records pointing at it. Fresh calls produced CTR JSON in S3.
  • Account: hit AWS's new free-plan gate (Amazon Connect is blocked on it) and made a deliberate, cost-controlled upgrade to the paid plan — sign-up credits + the $1 billing alarm + same-session number release keep real exposure to pennies.

Why this design

Two separate landing paths, on purpose: audio is written by Connect's recording behavior; the CTR is streamed out independently. They share nothing but the ContactId — which is also embedded in the .wav filename — so the filename literally carries the join key the rest of the pipeline reconciles on. This is the README's "arrive separately, match by contact ID" made real.

Firehose over a Kinesis Data Stream for the CTR target: a Data Stream bills a per-shard-hour charge (~$11/mo if left running), while Firehose has no idle cost (pennies per GB) — the free-tier-friendly choice. Tradeoff accepted: CTR streaming is forward-only, so the first calls (made before streaming existed) weren't exported; a second round of calls was placed to capture CTRs.

How it works

                         ┌── recording behavior ──► S3  CallRecordings/…/<contactId>.wav
Connect (call ends) ─────┤
                         └── Data streaming ──► Firehose ──(60s buffer)──► S3  CTR/…/*.json
                                                                                   │
                                   both keyed by ContactId  ◄───────────────────────┘

A recording is only produced if the flow turns recording on before the call reaches an agent (it captures the live audio leg) — turning it on after the transfer records nothing. Connect never writes CTRs to S3 directly; it PutRecords into Firehose, which owns the S3 write. That indirection creates two least-privilege IAM hops: Firehose→S3 (s3:PutObject on the bucket) and Connect→Firehose (firehose:PutRecord on the stream).

New concepts learned

Kinesis Firehose: a managed buffered writer to S3 — collect records, flush on a size/time hint. Like batching log lines and flushing every N seconds instead of one tiny write per event.
Recording-behavior placement: Connect only records from the moment the behavior block runs, so it must sit on the path before transfer-to-queue, and the flow must be published (saving a draft does nothing for live calls).
Forward-only streaming: enabling CTR streaming exports contacts created after it's on; existing contacts stay in Connect (visible in Contact search, retained ~24 months) but are never back-filled to S3.
Real PII surfaces here: a CTR's CustomerEndpoint.Address is the caller's actual phone number — the concrete field redaction must catch before any persistence or logging.
AWS free-plan vs paid-plan (2025 model): new accounts start on a credit-capped free plan that blocks cost-bearing services like Connect; the paid plan unlocks them while still honoring free-tier allowances and sign-up credits.

Interview talking point

"The source emits audio and metadata on two independent paths — the recording is written by the contact flow, the contact trace record is streamed through a Kinesis Firehose to S3 — and they're reconciled downstream purely by contact ID, which is even embedded in the recording's filename. I used Firehose rather than a Kinesis Data Stream specifically to avoid a standing per-shard-hour charge, and I treated the CTR's customer phone number as real PII to be redacted, because it is."

Phase 0 · Design & setup

The schema is the spine: modeling four entities around the Connect contact ID

2026-06-10

What was built

  • Repo skeleton: src/{connector,api,processing,emission,common}, sql/, infra/aws/, tests/, docs/.
  • sql/schema.sql — the four core tables (interactions, media_records, transcripts, signals) with inline rationale, constraints, indexes, and an updated_at trigger.
  • .gitignore + .env.example — secret-safety guardrails before any code that could leak one exists.
  • README.md — architecture diagram, data-model summary, reliability and HIPAA/BAA posture.

Why this design

Audio (S3) and metadata (contact trace records) arrive separately and out of order. The central decision was to make the Connect contact_id the business key on every table, so the two halves can be reconciled after the fact instead of assuming one combined payload. interactions is an anchor row that whichever path arrives first creates and the second enriches (upsert).

Alternatives considered: (a) a single wide "calls" table — rejected, it forces audio and metadata to arrive together; (b) a natural-key-only PK on contact_id — rejected in favor of surrogate id + UNIQUE(contact_id) for stable internal references plus a clean dedup key.

How it works

interactions (1) ──< media_records (1) ──< transcripts (1) ──< signals (many)
     contact_id          idempotency_key         redacted_text       signal_type
                         (UNIQUE → no
                          double-ingest)

Idempotency is enforced by the database: media_records.idempotency_key is UNIQUE, so a replayed recording fails the INSERT rather than relying on the app to remember to check. The status column (discovered → queued → ingested → transcribed → failed) is the pipeline's state machine and the basis for every monitoring query.

New concepts learned

Contact Trace Record (CTR): Amazon Connect's metadata record for a contact — queue, agent, timestamps, disconnect reason — emitted separately from the audio. Like a webhook event describing the call, arriving on its own channel.
Database-enforced idempotency: a UNIQUE constraint turns "don't double-ingest" from a hopeful app check into a guarantee the database keeps, even under retries and replayed queue messages.
Redact-before-persist: raw transcript text is held only in memory long enough to redact; only redacted_text is written. raw_text stays NULL, documenting the production design (encrypted-at-rest, access-restricted).

Interview talking point

"Recordings and metadata arrive on separate channels and out of order, so I keyed the whole model on the Connect contact ID and made the interaction row an upsert anchor — whichever side lands first creates it, the second enriches it. Idempotency is a UNIQUE constraint on the media record, so a replayed recording is rejected by the database, not by application logic I have to trust."
